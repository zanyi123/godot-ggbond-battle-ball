# 新子项目/新功能开发技能

## 触发条件

当用户提出以下类型需求时，本技能激活：

- 添加新功能、新系统
- 开发全新玩法模块
- 创建新的游戏机制
- 重构某个子系统为独立模块
- 添加 UI 系统优化
- 开发技能系统、装备系统等扩展
- 开发独立的 demo 或原型

## 核心原则

### 模块化优先
- 新功能必须先作为独立模块开发
- 独立 demo 验证后再接入主项目
- 不直接在主项目堆代码

### 修改范围控制
- 每轮最多修改 1-3 个文件
- 修改前必须说明计划
- 不得擅自修改公共接口
- 不得擅自修改节点名、信号名、输入映射

### 测试驱动
- 每个功能必须有测试编号
- bug 维修前先判断根因
- 每轮只修一个根因
- 输出复测清单

### 文档维护
- 模块需要维护：README_AI.md、CONTRACT.md、TEST_MATRIX.md、BUG_LEDGER.md
- 重要修改后更新文档
- 文档用于减少重复解释、限制 AI 乱改

## 工作流程

### 阶段判断

```
用户提出需求
  ↓
判断：是否需要独立模块？
  ├─ 否 → 直接实现（修改范围≤3文件）
  └─ 是 → 进入模块化开发流程
         ↓
     模块边界定义
         ↓
     文档创建（README_AI.md + CONTRACT.md）
         ↓
     独立 demo 规划
         ↓
     第一阶段最小目标
         ↓
     demo 实现 + 验证
         ↓
     TEST_MATRIX.md 测试
         ↓
     接入主项目（如需要）
```

### 判断是否需要独立模块

**需要独立模块的情况：**
- 新增完整系统（技能系统、装备系统、招募系统等）
- 重构某子系统为独立可测试模块
- 需要复杂 UI + 逻辑的多层系统
- 可独立运行验证的功能

**不需要独立模块的情况：**
- 简单的单文件逻辑修改
- 现有模块内部的小优化
- bug 修复（一般不触发模块化）

## 模块文档规范

### README_AI.md（模块上下文）

必须包含：
- 模块名称
- 模块目标
- 当前阶段
- 当前已完成内容
- 当前未完成内容
- 关键文件
- 关键约定
- 已知风险
- 下一步最小行动

**规则：**
- 只记录事实
- 内容保持简短
- 不写冗长解释
- 每轮重要修改后更新
- 更新时只输出需要替换或追加的段落

### CONTRACT.md（接口边界）

必须包含：
- 模块职责
- 模块不负责的内容
- 文件职责
- 公共接口
- 信号约定
- 节点约定
- 禁止事项
- 接入约定

**规则：**
- AI 必须遵守 CONTRACT.md
- 不得擅自违反
- 如必须违反，先说明原因、影响范围、替代方案、是否需要用户确认
- 公共接口、信号、节点命名、输入映射不得随意改动

### TEST_MATRIX.md（测试表）

每条测试必须包含：
- 测试编号
- 测试目标
- 操作步骤
- 预期结果
- 实际结果
- 状态

**测试状态：**
- 未测
- 通过
- 失败
- 阻塞

**规则：**
- 每个功能必须有测试编号
- 每个 bug 必须关联测试编号
- 修复后必须指定复测编号
- 不得只说"测试一下"，必须给明确复测项

### BUG_LEDGER.md（Bug 账本）

每个 bug 必须包含：
- bug 编号
- 标题
- 现象
- 复现步骤
- 预期结果
- 实际结果
- 完整报错
- 相关测试编号
- 严重程度
- 疑似相关文件
- 当前状态

**Bug 状态：**
- 未分析
- 待补充
- 待修复
- 已修复
- 已复测
- 暂缓

## 新功能处理流程

用户提出新功能时：

1. 判断当前阶段
2. 判断是否需要独立模块
3. 如需要，定义模块边界
4. 说明需要维护的文档
5. 规划独立 demo
6. 定义第一阶段最小目标
7. 列出需要用户提供的上下文
8. 给出下一步最小行动

**不得在新功能阶段顺手完成：**
- UI 完整化
- 存档接入
- 主项目接入
- 大规模重构
- 额外玩法扩展
- 无关系统优化

## 信息不足时的行为

停止写代码，只输出：
1. 当前无法判断的原因
2. 需要补充的文件
3. 需要补充的节点树
4. 需要补充的报错
5. 需要补充的代码片段
6. 用户从 Godot 哪里能找到这些信息
7. 补充后下一步会做什么

**不得：**
- 编造项目结构
- 假设节点存在
- 猜测完整实现代码

## 主项目接入规则

接入前必须满足：
- demo 能独立运行
- 主要测试项通过
- CONTRACT.md 稳定
- 公共接口明确
- 已知风险记录在 README_AI.md
- 接入方案明确
- 回滚方案明确

**接入清单：**
1. 需要修改的主项目文件
2. 每个文件的修改原因
3. 最小接入点
4. 接入顺序
5. 回滚方案
6. 接入后测试清单
7. 可能影响的旧系统
8. 是否需要用户确认

## 维修规则

维修规划必须包含：
- bug 分组
- 疑似共同根因
- 独立 bug
- 推荐修复顺序
- 本轮只修的根因
- 需要查看的文件
- 最小修改范围
- 不建议修改的文件
- 风险
- 复测编号

**维修补丁：**
- 只修一个根因
- 修改文件不超过 3 个
- 不违反 CONTRACT.md
- 不改无关系统
- 不添加新功能
- 不破坏已通过测试项
- 不擅自接入主项目
