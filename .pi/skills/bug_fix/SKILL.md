# 决竞球 Bug 维修 Skill

## 触发条件

当用户提出以下类型需求时，本技能激活：

- 游戏运行报错（Godot 编辑器报错、运行时崩溃）
- 功能异常（功能未按预期工作）
- 性能问题（帧率低、卡顿）
- 视觉问题（显示错乱、UI 异常）
- 物理问题（碰撞异常、穿墙）
- AI 行为异常（不动、卡死、错误行为）
- 数据丢失/错误（得分、状态保存等）

---

## 核心流程

```
用户报告问题
  ↓
读取：完整报错、报错行附近代码、节点树、最近改动、复现步骤
  ↓
用人话解释报错含义
  ↓
定位：报错文件、行号、对象、调用栈
  ↓
判断是否缺上下文（不足只列补充项，不猜）
  ↓
分析根因：区分表象 bug 和共同根因
  ↓
每轮只修一个根因，最多改 1-3 文件
  ↓
不重构无关代码，不改公共接口、节点名、信号名、输入映射
  ↓
输出最小补丁或替换函数
  ↓
给复测步骤和风险
  ↓
修完更新 bug 状态
```

---

## 规则约束

### 读取优先级

维修前必须先读取（按顺序）：

1. **完整报错**：Godot 编辑器完整报错信息、运行时错误日志
2. **报错行附近代码**：报错行 ±10 行的完整代码
3. **节点树**：相关节点的层级关系、脚本挂载位置
4. **最近改动**：最近修改的文件列表
5. **复现步骤**：如何触发问题

### 信息不足时的行为

**不得：**
- 猜测实现代码
- 假设节点存在
- 假设脚本挂载正确
- 假设信号已连接
- 假设公共接口存在
- 编造项目结构

**只输出：**
1. 当前无法判断的原因
2. 需要补充的文件
3. 需要补充的节点树
4. 需要补充的报错
5. 需要补充的代码片段
6. 用户从 Godot 哪里能找到这些信息
7. 补充后下一步会做什么

### 维修规划原则

维修规划必须包含：

- **bug 分组**：哪些 bug 同源
- **疑似共同根因**：最可能的原因
- **独立 bug**：单独处理
- **推荐修复顺序**：先修根因
- **本轮只修的根因**：只修一个
- **需要查看的文件**：明确文件列表
- **最小修改范围**：只改必要部分
- **不建议修改的文件**：避免范围扩大
- **风险**：可能的副作用
- **复测编号**：修复后验证项

### 补丁输出原则

**允许：**
- 只输出需要替换的函数
- 输出最小 diff
- 输出单个完整文件
- 输出明确的新文件内容

**禁止：**
- 输出无关代码
- 输出未请求文件
- 输出大量完整项目代码
- 改变公共接口（未说明原因）
- 重构无关代码
- 混入额外功能

### 修改范围限制

**每轮最多：**
- 修改 1-3 个文件
- 修复 1 个根因

**禁止：**
- 擅自修改公共接口
- 擅自修改节点名
- 擅自修改信号名
- 擅自修改输入映射
- 重构无关代码
- 顺手完成额外功能

## 常见问题检查规则

### Nil / null instance

优先检查：

- 节点路径是否正确
- `@onready` 是否成功获取节点
- 导出变量是否已赋值
- 场景切换后引用是否失效
- 信号传入对象是否为空
- 资源是否加载成功
- 脚本是否挂在正确节点上

### Node not found

优先检查：

- 代码中的节点路径
- 实际 Scene 树
- 节点是否改名
- 层级是否变化
- 脚本挂载位置是否正确

### Invalid call

优先检查：

- 调用对象真实类型
- 方法是否存在
- 脚本是否挂载
- 信号参数是否正确
- 是否把子节点当成主体对象使用

### 信号问题

优先检查：

- 信号是否连接
- 是否重复连接
- 信号参数是否匹配
- 回调函数名是否正确
- 节点生命周期是否早于连接时机
- 场景切换后引用是否失效

### 动画和状态机问题

优先检查：

- 状态列表
- 状态优先级
- 状态切换入口
- 动画名是否一致
- `animation_finished` 是否触发
- `_process` 或 `_physics_process` 是否同时改状态
- 是否多个系统同时控制动画
- 是否缺少不可打断状态
- 是否缺少死亡、受伤、攻击等状态锁定

**推荐状态优先级：**

`death > hurt > attack > dodge > jump/fall > run/idle`

### 物理问题

优先检查：

- `collision_layer` / `collision_mask` 设置
- `move_and_slide` 与碰撞层匹配
- `is_defeated` 状态下的碰撞层
- `set_physics_process` 是否被禁用
- 隔离墙的碰撞层和位置

## 项目特殊检查清单

### 决竞球项目特有检查

1. **AI 相关报错**
   - `ai_manager` 的 `_is_valid` 检查
   - `ai_players` 数组是否为空
   - `ball_node` / `input_manager` / `battle_manager` 引用是否有效
   - `is_defeated` 状态下的逻辑

2. **外场隔离墙**
   - `penalty_walls` 是否为 null
   - 球员 `is_penalized` 状态
   - `collision_mask` 是否包含 layer 5 (16)
   - 隔离墙位置是否正确

3. **球员状态**
   - `is_defeated` = true 后不应再被球击中
   - `collision_layer` 被击败后 = 0，传送完成后恢复
   - `set_physics_process` 传送期间 = false

4. **通信系统**
   - `comm_system` 引用是否有效
   - `message_sent` 信号是否正确连接
   - `record_message` 是否正常写入

5. **数据加载**
   - `DataManager.get_character_by_id`
   - `characters.json` 格式正确性
   - `skills.json` 格式正确性

6. **场地系统**
   - `field_zone` 边界常量与实际绘制是否一致
   - `凹字形` 缺口坐标
   - `_clamp_to_outer_field` 凹字形感知

- **不得**一次修复多个根因

- **不得**混入额外功能优化
