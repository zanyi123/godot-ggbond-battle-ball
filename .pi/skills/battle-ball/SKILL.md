# 决竞球 (Battle Ball) 项目开发技能

本技能用于规范「决竞球 Battle Ball」Godot 项目的 AI 辅助开发流程。

## 核心工作原则

### 1. 消歧义
收到用户消息后，**先思考是否存在歧义或不明确之处**：
- 如果有歧义，**必须先提问确认**，不要猜测后直接执行
- 如果需求清晰无歧义，直接进入下一步

### 2. 复述与纲要
回答时**必须先做两件事**：
- **复述理解**：用简短的话重述用户的需求，确认双方理解一致
- **给出纲要**：列出解决问题/实现功能的步骤大纲，让用户心中有数

### 3. 过程精简
- 解决问题的**详细过程不要全部显示在对话框里**（如大量的中间文件内容、逐步调试日志等）
- 只展示关键节点和关键决策
- 需要用户确认的重要节点除外

### 4. 完成总结
每项工作完成后，**必须总结**：
- 做了什么（完成了哪些要点）
- 改动涉及哪些文件
- 是否有待后续处理的事项

### 5. 自检自测
对于新功能或功能扩展：
- **实现后必须自检**：检查代码逻辑、节点引用、信号连接等是否正确
- **必须进行可运行的测试**：通过运行场景、打印日志等方式验证功能
- 如果发现自测无法覆盖（如需要手动操作验证），**明确告知用户需要手动验证的内容**

### 6. 有疑问就问
- 遇到不确定的设计决策、数值设定、交互方式等，**主动提问**，不要自行假设
- 宁可多问一句，也不要做错返工

## 技术栈

- **引擎**：Godot 4.x（以项目实际版本为准）
- **语言**：GDScript
- **项目路径**：`E:/项目储存/决竞球battle-ball/`

## 开发规范

- 遵循 Godot 4 最佳实践（场景化设计、信号驱动、节点组合优于继承）
- 代码需有清晰的中文或英文注释
- 变量/函数命名遵循 GDScript 风格指南（snake_case）
- 合理组织项目目录结构

## 工作流

```
收到需求
  → 思考是否有歧义（有则先提问）
  → 复述理解 + 给出纲要
  → 精简执行（关键节点展示）
  → 自检自测
  → 完成总结
```
